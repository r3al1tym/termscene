Showcase fidelity audit

termscene · the 8 named-terminal examples in SHOWCASE_INDEX, checked against each tool's real on-screen conventions.

2026-06-17 · sources are primary (tool source repos, shipped theme files, official docs) · one research pass per tool

Bottom line. Every example is the right flavor, but none is pixel-faithful — each has 2–4 tells a user of that tool would catch instantly. The misses split cleanly: most are data edits to scenes-data.ts / themes.ts I can apply now; a recurring set are engine gaps — the renderer's prompt is a single-line, single-color string, so it can't yet do split-color prompts (Ubuntu, p10k), two-line prompts (p10k, Starship), persistent footers (Gemini), or Warp's command Blocks. Those four are the least faithful precisely because their identity lives in the prompt/chrome the engine can't express.

Warp low

Gets mac chrome + rounded window right; misses all three signals that say "Warp".

wrongNo Blocks eng

Warp's defining visual: each command + its output is a discrete block, divider lines between, a failed command gets a red left-bar + red-tinted bg. Fix: render each step as a block — divider, left gutter, red accent for non-zero exit. This is the #1 change to make it read as Warp.

What this means for the fix

Two distinct jobs. Data-layer (most findings) — corrected glyphs, output formats, model ids, banner art, and theme hexes I can apply to scenes-data.ts and themes.ts in one pass; macOS, Ubuntu, Claude Code, Codex, and Gemini reach high fidelity this way. Engine-layer — the renderer's single-line / single-color prompt is the shared blocker behind the four weakest examples: p10k and Starship need a two-line prompt with a colored ❯; Ubuntu and p10k need per-segment prompt colors; Gemini wants a persistent footer; Warp wants command Blocks. These are the higher-leverage frontier work — they'd lift the whole showcase, not just one tile.
